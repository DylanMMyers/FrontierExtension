impl notes (post coding):

I noticed that when using the triadic constraint, you end up with significantly more lambda values in 
the returned set (post further testing Dylan here, this actually ends up being not true). I believe this occurs because since the triadic constraint is weaker than the raw 
triangle inequality constraint, our ORLP dual has "less idea" of how far it can preturb the original solution. 
This results in it pushing the boundary less, which results in smaller right bounds and in turn more values used 
overall. This is the main tradeoff I think, there might be more nuanced things underlying but this is the main idea.

I think similarly to the main tradeoff between the original LambdaCC issue of solving optimally vs. number of 
breakpoints, using the wedge inequality increases the amount of clusterings we have to go through to cover 
our entire range with a lower overall runtime in contrast to the triangle inequality method which has higher runtime 
but goes through less iterations. For very large graphs, only the open wedge constraint is really feasible runtime 
wise.

For small graphs, large epsi (1 or close to it) need very few solutions to cover the range

Might be a skill issue, but it took 30 minutes to run a graph with 198 nodes (jazzA graph). I think I made 
it pretty fast but maybe there are some more novel speedups which could be made. At this point, though, I feel 
like mostly optimizations to squeeze out runtime are available.
